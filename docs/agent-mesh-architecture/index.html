Agent Mesh 架构文档

架构分层

┌────────────────────────────────────────────┐
│          OpenClaw (编排层)                   │
│  管道编排 · 状态管理 · 失败接管 · 报告输出    │
├────────────────────────────────────────────┤
│          MCP Bridge (通信层)                │
│  agent-chat · agent-delegate · context-r/w │
├────────────────────────────────────────────┤
│       Agent Messenger (消息总线 v2)          │
│  文件消息队列 · 主题广播 · 线程追踪 · 自动发现 │
├────────────────────────────────────────────┤
│    Agent 层                                │
│  OpenClaw | Hermes | QoderCLI              │
│  with Architect/Researcher/Planner/...     │
└────────────────────────────────────────────┘

Agent 角色

角色后端职责
OrchestratorOpenClaw管道编排、阶段推进、报告输出
HermesDeepSeek API需求分析、排期、测试用例、测试执行
ArchitectQoderCLI (ingeek-sw-arch)系统架构、模块边界、接口定义
ResearcherQoderCLI (ingeek-sw-reviewer)代码探索、事实发现、差距检测
PlannerQoderCLI (ingeek-sw-arch)任务分解、排期细化、合同定义
ImplementerQoderCLI (ingeek-sw-dev)TDD 编码实现
TesterQoderCLI (ingeek-sw-test)单元测试、覆盖率、静态分析
ReviewerQoderCLI (ingeek-sw-reviewer)代码审查、质量门禁、架构合规

通信协议

方式 1：MCP Bridge — Orchestrator → Agent

OpenClaw ──agent-chat──▶ Hermes        (同步 <8s)
OpenClaw ──agent-delegate──▶ QoderCLI  (异步，返回 task_id)

方式 2：消息总线 — Agent ↔ Agent

Architect ──send──▶ Planner: "设计评审请求"
Planner   ──reply──▶ Architect: "评审完成"
Implementer ──broadcast──▶ 所有订阅者: "代码待审查"

边界规则

  1. OpenClaw 只编排，不替 Agent 写代码
  2. Agent 之间直接沟通——不经过 OpenClaw 中转
  3. 每阶段输出必须有验证
  4. 生成与审查必须分离（不同 Agent）
  5. 架构变更必须经 Architect 评审

失败处理

机制参数
每阶段重试最多 3 次
超时（同步）8 秒
超时（异步）300 秒
3 次全失败OpenClaw 接管，输出异常报告
