Agent Mesh 架构

📅 2026-05-26 📖 阅读约 5 分钟
架构Agent Mesh设计MCP

Agent Mesh 是一个跨 Agent 协同网络的架构设计方案。本文档从全局视角描述系统的组件关系、数据流和设计决策。

架构目标

  • 多 Agent 互联 — 任意 Agent 之间可以互相发现和通信
  • 异步优先 — 耗时任务非阻塞，通过任务队列管理
  • 统一上下文 — 共享上下文空间，避免信息孤岛
  • 热插拔 — Agent 可以随时加入或离开网络

核心组件

1. MCP Bridge v2（消息路由层）

作为整个 Mesh 网络的中心枢纽，负责：

  • 工具注册 — 每个 Agent 向 Bridge 注册自己的能力（工具列表）
  • 消息路由 — 根据目标 Agent 名称，将请求转发到正确的目的地
  • 任务队列 — 异步任务的管理和状态追踪
  • 上下文存储 — 共享上下文的读写接口

2. Agent 节点

  • OpenClaw — 前端入口 Agent，处理用户交互和任务分发
  • Hermes — 主力工作 Agent，执行搜索、代码、文件操作
  • QoderCLI — 工具型 Agent，提供系统级操作和脚本执行

3. 共享工作区（Workspace）

文件系统中的共享目录，Bridge 进程负责管理：

~/.openclaw/mcp-bridge/workspace/
├── context.json         # 共享上下文（键值存储）
└── tasks/               # 异步任务结果
    ├── task_abc123.json
    └── task_def456.json

通信协议

所有通信通过 MCP (Model Context Protocol) 进行，采用 JSON-RPC 格式：

// 请求
{
  "jsonrpc": "2.0",
  "method": "tools/call",
  "params": {
    "name": "agent-chat",
    "arguments": {
      "agent": "hermes",
      "message": "分析这份代码"
    }
  },
  "id": 1
}

// 响应
{
  "jsonrpc": "2.0",
  "result": {
    "content": [{ "type": "text", "text": "分析结果..." }]
  },
  "id": 1
}

数据流设计

三种主要的通信模式：

同步调用

适用于快问快答（≤8 秒）。调用方 Agent 发送请求后阻塞等待，Bridge 转发并聚合响应后返回。

异步委派

适用于耗时任务。调用方立即收到 task_id，后续通过 task-result 轮询结果。

Agent A → Bridge: agent-delegate(hermes, "分析日志")
Bridge → Agent A: { task_id: "abc123" }

Agent A → Bridge: task-result("abc123")
Bridge → Agent A: { status: "completed", result: "..." }

上下文共享

通过 context-write/context-read 实现跨 Agent 的信息传递。数据持久化在 context.json 中。

设计决策记录

决策方案理由
通信协议MCP + JSON-RPC标准化、与现有工具兼容、易扩展
状态存储本地文件 JSON零外部依赖、易调试、性能足够
任务模型同步+异步双模灵活性，快任务不引入异步开销
发现机制Bridge 中心化注册简单可靠，适合 Agent 数量不多的场景
