Embedded Verify Helpers 模式

用 static helper 消除 N 处重复的读+比较循环

问题

在嵌入式 Flash 驱动（或类似的外设驱动）中，常有多个独立的操作步骤需要进行数据校验：

  • Erase verify：读 256B 出来，按 uint32 word 检查是否全为 0xFFFFFFFF
  • Write verify：读 256B 出来，逐字节比对 WriteBuffer 中的期望数据
  • Compare handler：读一个 chunk，逐字节比对 CallerBuffer 中的数据

这三处代码基本一样：读 flash → 比较 → 返回 OK/FAIL。区别仅在比较模式和期望数据来源。如果独立实现，每处都是 14-26 行相似的代码，总共 58+ 行重复。

更隐蔽的风险：每处独立实现都可能引入不同的 Bug（比如 readBuf 类型不一致、长度判断漏了、未对齐访问）。DRY 原则在这里不仅是文风问题，是正确性问题。

解决方案

提取一个 static helper 函数，用 enum 模式区分 ERASE/BYTE 两种比较方式：

/* ========== 模式枚举（.c 内，不暴露） ========== */ typedef enum { EXT_FLS_VERIFY_ERASE, /* Expect all bytes 0xFF */ EXT_FLS_VERIFY_BYTE /* Compare byte-by-byte against ExpectedPtr */ } ExtFls_VerifyModeType; /* ========== Helper 函数 ========== */ static Std_ReturnType ExtFls_VerifyChunk( uint32 Address, ExtFls_LengthType Length, const uint8 *Expected, ExtFls_VerifyModeType Mode ) { uint32 readBuf[64U]; /* 256 bytes, 4-byte aligned */ uint16 i; if (Length > 256U) return E_NOT_OK; if ((Mode == EXT_FLS_VERIFY_ERASE) && ((Length & 3U) != 0U)) return E_NOT_OK; /* P2: ERASE mode needs word alignment */ if (ExtFlashDriver_Read(Address, Length, (uint8 *)readBuf) != E_OK) return E_NOT_OK; if (Mode == EXT_FLS_VERIFY_ERASE) { uint32 *wordPtr = readBuf; /* uint32[] → no alignment issue */ uint16 n = (uint16)(Length / sizeof(uint32)); for (i = 0; i < n; i++) if (wordPtr[i] != 0xFFFFFFFFU) return E_NOT_OK; } else { for (i = 0; i < (uint16)Length; i++) if (((uint8 *)readBuf)[i] != Expected[i]) return E_NOT_OK; } return E_OK; }

调用方示例

三处调用全都是同一个模式：

/* Erase verify — 传 NULL，ERASE 模式不走 Expected */ if (ExtFls_VerifyChunk(Address, 256U, NULL, EXT_FLS_VERIFY_ERASE) != E_OK) { /* handle error */ } /* Write verify — 按偏移定位到 WriteBuffer 中的预期字节 */ if (ExtFls_VerifyChunk(Address, readSize, &WriteBufferPtr[VerifyAddress - WriteAddress], EXT_FLS_VERIFY_BYTE) != E_OK) { /* handle error */ } /* Compare handler — 逐字节比对 ReadBuffer 中的数据 */ if (ExtFls_VerifyChunk(Address, ReadCount, ReadBufferPtr, EXT_FLS_VERIFY_BYTE) != E_OK) { /* data mismatch or read failure */ }

关键设计决策

决策方案理由
函数位置 CDD 层 static 不是 HAL 接口，不跨文件调用；static 表示「这里是实现细节，调用方自己处理状态管理」
Buffer 类型 uint32[64] 保证 4 字节对齐，ERASE mode 强转 uint32* 无 HardFault 风险
模式分离 enum 参数 比函数指针回调简单，比两个独立函数 DRY
不接管状态 只做 chunk 级操作 地址推进、剩余长度管理、回调通知全在调用方。helper 只负责「读+比」，单一职责
ERASE 传 NULL ERASE mode 不解引用 Expected C 标准安全（条件分支互斥），不用造空数组。BYTE mode 必须传有效指针

陷阱 & 教训

P0: Buffer 对齐
uint8[256] 栈数组只保证 1 字节对齐，ARM Cortex-M 上强转 uint32* 后， 如果 CCR.UNALIGN_TRP=1 会 HardFault。 声明为 uint32[64] 从根源解决。
P1: 旧代码的隐藏 UB
审查发现原 Compare handler 用的 uint32[16] = 64 bytes 作为 readBuf。 如果 FlsFastRead 配置 > 64 字节，旧代码是栈溢出 UB（不出错是因为凑巧够用）。
技巧：小抠抓出的嵌入知识差距
栈对齐问题在 x86 开发环境完全感知不到，只有 ARM Cortex-M 才会暴露。 这类「跨架构盲区」正是代码审查中嵌入专家（小抠）的价值所在。

适用场景

  • 驱动层有 N 个操作都需要「读一个 chunk → 检查内容」
  • 比较模式固定（全擦除 0xFF、逐字节比对待定数据、mask 模式等）
  • 调用方自己管理状态机（地址、剩余长度、回调）
  • buffer 大小由内部决定（通常 256B，合适 SPI Flash page 对齐）

不适用场景

  • 需要跨模块共享（应该放在 HAL 层，函数指针回调注入比较器）
  • 比较逻辑需要动态注入（应该用回调函数指针，不要用 enum + if-else）
  • buffer 大小取决于调用方配置（应该把 buffer 也作为参数传给 helper）
