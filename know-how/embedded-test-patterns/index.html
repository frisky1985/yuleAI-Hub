嵌入式 PC 端测试模式

NP 平台实践中提炼的可复用模式与规则

一、架构概述

嵌入式 MCU 软件无硬件依赖测试的核心挑战：所有硬件寄存器、中断、RTOS 调用必须在 PC 端模拟。NP 平台的实践中总结了 6 种测试模式。

二、六大测试模式

模式 1: 回调注入 (Callback Injection)

Wdg

将时间源、硬件访问等外部依赖通过函数指针注入，测试使用 mock 回调。

// Wdg.h — 外部时间源通过回调注入
void Wdg_Init(const Wdg_ConfigType* configs, uint8_t count,
              uint32_t (*getTimeMs)(void));
// 测试: mock 时间控制
static uint32_t mock_time_ms = 0;
static uint32_t Mock_GetTimeMs(void) { return mock_time_ms; }

模式 2: 函数表 Mock (Function Table / VTable)

HAL Mock

用函数指针 struct 替代直接硬件调用。MCU 实现和 PC mock 通过同一接口契约切换。

typedef struct {
    void (*Init)(void);
    uint32_t (*GetSystemClock)(void);
    void (*PerformReset)(void);
} Mcu_HalOps;

extern const Mcu_HalOps Mcu_Hal;  // 链接时选择实现

模式 3: 手动 Mock + Call Counter (Manual Mocking)

MemMgr, StateM

手工编写 mock 头文件和实现，每个 mock 函数维护 callCount。比 CMock 轻量，无 Ruby 依赖。

// mock_config.c
uint32_t Mock_CheckEvent_callCount = 0;
uint32_t Mock_ActiveAction_flag = 0;

// 测试: 验证回调是否被正确调用
StateM_MainFunction();
TEST_ASSERT(Mock_CheckEvent_callCount > 0);

模式 4: 属性擦除 (Attribute Stripping)

pc_test_fix.h

嵌入式 __attribute__((section("..."))) 在 Mach-O 上不兼容。通过 -include pc_test_fix.h 全局擦除。

// pc_test_fix.h — force-included before all sources
#pragma clang diagnostic ignored "-Wattributes"
#define __attribute__(x)  /* empty */

模式 5: 累积状态容忍 (State Accumulation Tolerance)

StateM

Unity 测试共享进程空间，静态全局变量会跨 case 累积。使用行为级断言而非绝对计数值。

// 坏的: 依赖绝对计数值
for (int i=0; i<9; i++) {
    StateM_MainFunction();
    TEST_ASSERT(status & bit);  // 计时器跨 case 累积
}

// 好的: 行为级断言
int active = 0;
for (int i=0; i<15; i++) {
    StateM_MainFunction();
    if (status & bit) active++;
}
TEST_ASSERT(active > 0);  // 至少有时活跃

模式 6: 构建时源码修正 (Build-Time Source Patching)

MemMgr (heap_4.c)

公司 SDK 内部有架构依赖 (如 portPOINTER_SIZE_TYPE uint32_t 在 ARM64 指针截断)，用 add_custom_command 在 CMake 构建时 sed 修正。

# CMakeLists.txt — 构建时生成修复版
add_custom_command(
    COMMAND sed 's/portPOINTER_SIZE_TYPE uint32_t/portPOINTER_SIZE_TYPE uintptr_t/'
    "${CMAKE_CURRENT_BINARY_DIR}/heap_4_fixed.c")

四、适用场景与准则

  • PC 端可测性：所有依赖必须通过接口注入，不能直接调用硬件寄存器
  • Mock 合约原则：mock 需记录 callCount，验证调用次数而非副作用
  • 零硬件假设：测试不得依赖特定 MCU 特性（如寄存器复位值）
  • 渐进重构：先加测试覆盖，再重构代码，不改行为
  • 编译双验证：PC 端功能测试 + ARM 端编译检查
