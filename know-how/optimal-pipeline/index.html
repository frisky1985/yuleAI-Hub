最优 Agent 流水线

将 OpenSpec 规范的 SDD、TDD 测试驱动和 Harness Engineering 自循环融合为单一最优流水线

SDD TDD Harness Engineering Agent Mesh OpenSpec

🎯 三方法结合

三个方法论各有所长，结合后形成完整的开发闭环：

📐 OpenSpec (SDD)

优势：规范优先，需求、场景、接口、质量四位一体

  • openspec/ 目录结构，specs 是源真值
  • Delta-based：每个 change 只记录差异
  • GIVEN/WHEN/THEN 场景驱动测试设计

🔄 TDD

优势：测试即契约，红/绿/重构循环保证质量

  • 测试从 spec 场景推导（而非开发者直觉）
  • RED → GREEN → REFACTOR 每个任务循环
  • 验证即回归测试集

🤖 Harness Engineering

优势：自主 ITR 循环，agent 直接对话

  • Implementer → Tester → Reviewer 自治循环
  • Message bus 文件收件箱，agent 间零中间人
  • Post-loop 自改进

🧠 OpenClaw

优势：编排层，注入上下文、管理子 agent

  • 8 阶段流水线编排
  • 所有 artifact 结构化管理
  • 最终摘要报告压缩输出

🏗️ 流水线总览

0 Init & Context 创建 OpenSpec change，设定目标，建立 agent 上下文 Orchestrator
1 Explore & Brainstorm 研究员探索代码库 → 架构师出 2-3 方案 → 推荐最优 Researcher → Architect
2 SDD Spec Authoring 需求 (FR/NFR) + GIVEN/WHEN/THEN 场景 + 接口定义 + 质量标准 Architect → Tester → Architect → Reviewer
3 Design & Contracts 模块级架构设计 + 每个模块的输入/输出/不变式契约 Architect
4 Task Decomposition 任务分解，每个任务标注覆盖的 spec 需求编号 Planner
5 SDD→TDD Implement 核心阶段：每个任务 SDD 推导测试 → TDD 红/绿/重构 → 质量门禁 Tester→Impl→Reviewer
6 Verification OpenSpec 三维验证：完整性、正确性、一致性 Reviewer
7 Sync & Archive spec deltas 合并到源真值，归档 change Orchestrator
8 Final Report 压缩式摘要：统计 + 验证结果 + 任务状态 Orchestrator

📐 核心创新：SDD→TDD 追溯链

传统 TDD 的测试来自开发者直觉。SDD→TDD 中，测试从 spec 场景推导：

需求 FR-003: "系统应在 200ms 内响应查询"
  ↓
场景: "用户查询存在的数据"
  GIVEN 数据库中有 100 条记录
  WHEN 用户查询"keyword"
  THEN 系统返回匹配结果，耗时 <200ms
  ↓
SDD 推导测试:
  Test: query_existing_data_performance
  Setup: 插入 100 条记录
  Input: "keyword"
  Expect: 返回结果 <200ms
  ↓
TDD RED: 写测试（预期失败，因为实现不存在）
TDD GREEN: 实现最小代码通过测试
TDD REFACTOR: 清理、保持 spec 合规

Agent 对话模式

Agent 之间通过文件收件箱直接对话，编排器只负责推进阶段：

# Architect 发送消息给 Researcher
python3 agent-messenger.py send architect researcher "探索代码库" \
  "请分析 xxx 模块的现有接口"

# Researcher 回复方向消息
python3 agent-messenger.py send researcher architect "探索结果" \
  "发现 xxx 模块缺少错误处理..."

# 编排器检查进度，推进到下一阶段
python3 harness-orchestrator.py phase spec

📊 关键技术指标

  • 追溯性：每个测试可追溯到 spec 场景，每个实现可追溯到需求编号
  • 并行度：无依赖的任务可并行执行（message bus 天然支持）
  • 自主度：Implementer→Tester→Reviewer 自治循环，编排器只在阶段间介入
  • 增量性：OpenSpec delta-based，只记录变化部分
  • 可验证：3D 验证（完整性/正确性/一致性）+ spec 驱动的质量标准

📖 使用方法

启动完整流水线

cd ~/.openclaw/mcp-bridge
python3 harness-orchestrator.py run "你的项目目标"

单阶段运行

python3 harness-orchestrator.py phase init "目标"
python3 harness-orchestrator.py phase explore
python3 harness-orchestrator.py phase spec
python3 harness-orchestrator.py phase design
python3 harness-orchestrator.py phase tasks
python3 harness-orchestrator.py phase implement
python3 harness-orchestrator.py phase verify
python3 harness-orchestrator.py phase archive
python3 harness-orchestrator.py phase report

查看状态

python3 harness-orchestrator.py status

重置（开始新 cycle）

python3 harness-orchestrator.py reset

📝 与之前 DevFlow 的关系

之前的 DevFlow 2.0（7 阶段 DevOps 流水线）依然可用，但本次升级将 Agent 协作从 编排器中介 升级为 Agent 直接对话（Harness Engineering + Superpowers），并引入 OpenSpec 作为规范层。

三者的关系：

  • OpenSpec = 规范层（需求→场景→接口→质量，spec is truth）
  • SDD→TDD = 开发方法（从规范推导测试，红/绿/重构实现）
  • Harness Engineering = 执行层（agent 自主 ITR 循环，message bus 通信）
  • Agent Mesh = 基础设施（文件收件箱 + 消息路由 + QoderCLI agent 绑定）
