Agent Mesh 技能清单 + 消息契约

1. Orchestrator

运行于: OpenClaw   通信: agent-chat / agent-delegate

能力:

  • 管道编排：按阶段推进 Harness Engineering 管道
  • 状态管理：维护 orchestrator-state.json
  • 失败接管：3 次重试后自动介入修复
  • 报告输出：生成最终总结报告
python3 harness-orchestrator.py run "<goal>"
python3 harness-orchestrator.py resume   # 续传
python3 harness-orchestrator.py status   # 状态

2. Hermes

运行于: DeepSeek API（Hermes CLI）   通信: agent-chat / agent-delegate

能力:

  • 需求分析：解析用户需求，输出结构化需求文档
  • 排期：基于需求制定时间线和里程碑
  • 测试用例：为代码创建测试用例
  • 测试执行：运行测试并输出测试报告
  • 快速代码审查
hermes -z "prompt" --yolo              # 同步（agent-chat）
hermes -z "long task" --yolo           # 异步（agent-delegate）

3. QoderCLI

运行于: DeepSeek API（QoderCLI）   通信: agent-delegate

能力: 架构设计、编码实现、代码审查、代码探索

子角色（6 个 QoderCLI Persona）

角色Qoder Agent职责
Architectingeek-sw-arch系统架构、模块边界、接口定义
Researcheringeek-sw-reviewer代码探索、差距分析、技术债务识别

qodercli -p --dangerously-skip-permissions "prompt"          # 默认
qodercli -a ingeek-sw-dev -p "implement this"               # 指定子角色

Agent 间直接对话 — 消息总线

核心理念：Agent 从不直接发消息给对方——而是写入对方的收件箱文件。对方自读。写文件是原子操作，不会丢消息。不依赖网络、不依赖两者同时在线。

消息总线目录结构

~/.openclaw/mcp-bridge/workspace/messages/
  ├── queue/                 # 活跃消息收件箱
  │   ├── architect/         ← 每个角色有独立收件箱
  │   ├── researcher/
  │   ├── planner/           ← Planner 的待读消息
  │   └── ...
  ├── archive/               # 过期消息（30分钟自动移入）
  ├── topics/                # 广播主题存档
  ├── threads/               # 对话线程追踪
  └── subscriptions.json     # 主题订阅表

命令参考

命令说明示例
send <from> <to> <subj>点对点发送消息send architect planner "design-review"
read <role> [--last N]读取收件箱read planner --last 5
reply <msg_id> <from>回复消息（自动填充 in_reply_to）reply abc123 planner
wait <role> [--timeout 60]阻塞等待新消息（轮询 1s 间隔）wait reviewer --timeout 120
broadcast <from> <topic> <subj>广播到主题（仅发送给订阅者）broadcast impl "code/ready" "done"
subscribe <role> <topic>订阅主题subscribe tester "code/ready"
unsubscribe <role> <topic>退订主题unsubscribe tester "code/ready"
thread <msg_id>查看对话线程thread abc123
discover发现所有可用 Agentdiscover
status消息总线状态（各收件箱队列长度）status
purge [--age 86400]清理过期消息（默认 24h）purge --age 3600

对话流程（点对点）

Agent A                          Agent B
  │                                │
  │  1. send(role, subject)        │
  │───────────────────────────────▶│  (写入 B 的收件箱文件)
  │                                │
  │  2. (B 在做其他任务)            │
  │                                │
  │  3. B 空闲时 poll              │
  │     messenger.py wait          │
  │                                │
  │  4. 读到新消息                  │
  │                                │
  │  5. reply(msg_id, body)        │
  │◀───────────────────────────────│  (写入 A 的收件箱文件)
  │                                │
  │  6. A 下次 read 看到回复        │

不丢消息的保证

机制说明
原子写先把消息序列化为完整 JSON，再写入收件箱文件。不会写一半
自动清理消息在收件箱超过 30 分钟自动移入 archive，不丢失不堆积
存档保留archive 目录永久保留消息体，purge 默认只清理 24h 以上
契约校验每条消息带 contract: "message-v1"，Agent 可拒绝格式不合规的消息
发现机制discover 自动列出所有在线 Agent，发错角色名会报错

一键验证

# 1. 查看所有 Agent 在线状态
python3 agent-messenger.py discover

# 2. 查看消息总线状态
python3 agent-messenger.py status

# 3. 发送测试消息
python3 agent-messenger.py send architect planner "ping" '{"test": true}'

# 4. 读取收件箱
python3 agent-messenger.py read planner --last 1

# 5. 回复
python3 agent-messenger.py reply <msg_id> planner '{"pong": true}'

# 6. 查看完整对话线程
python3 agent-messenger.py thread <msg_id>
