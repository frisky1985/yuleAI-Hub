⚡ 设计驱动开发（DDD）工作流 — 文档先行，用户确认后编码

📅 2026-05-27 ⏱️ 方法论技能
DDD工作流方法论文档驱动

文档先行、代码次之的开发方法论。所有设计变更必须先更新文档、等待用户确认，然后才能实现。核心原则：先出文档，确认后再推进代码。

适用场景

  • 用户提出 UI 布局变更或设计修改时
  • 用户反馈需要代码改动时
  • 用户说"先出文档"或"先不要急着改代码"时
  • 迭代设计阶段
  • 用户提供中间反馈时

核心原则

每次提出修改意见时，必须先修改文档需求。如果中间有修改意见，也是先调整文档，等确认后再推进代码编写。绝对不能先改代码再汇报。

这条规则适用于所有反馈，即使是看起来微小的调整也不例外。

标准流程

User feedback → Update design document → Wait for confirmation → Implement code

Step 1: 停止并倾听

  • 不要触碰任何代码
  • 完整理解需求，必要时提出澄清问题
  • 确认你理解了需求

Step 2: 更新设计文档

  • 找到或创建相关的设计文档（docs/ 或 workspace 根目录）
  • 包含：变更内容、变更原因、视觉布局（ASCII 图）、受影响的组件列表
  • 在底部添加版本变更日志

Step 3: 提交确认

  • 总结更新的文档，高亮关键变更
  • 询问确认："确认后我推进代码"

Step 4: 确认后编码

  • 严格遵循文档，不 scope creep
  • 提交信息引用设计文档
  • 展示结果供验证

Brainstorming 批量化工作流

当用户说"brainstorming一下"或"给一些建议"：

  1. 快速审查当前状态，识别自然的下一个方向
  2. 将方向分组为 2-3 个方案（A/B/C），每个含 3-6 个子项、时间估算、说明
  3. 以表格形式呈现，让用户选择
  4. 选定方向后拆分为编号子项（A-1, A-2...），按复杂度排序
  5. 分批实施：首批快速落地 vs 第二批深度开发
  6. 按批次顺序执行，完成后询问下一个方向

确认后并行执行

当多任务计划获得确认后，立即将独立的工作流并行化：

  • 识别独立任务（不同组件、不同服务、不同页面）
  • 分组给 2-3 个子代理
  • 每个子代理需包含构建验证
  • 收集结果，修复集成问题，统一提交

⚠️ 常见陷阱

  • 编码前未更新文档是最严重的违规。 任何变更都不能被认为是"太小"而不需要文档
  • 实施中的反馈必须重新启动文档周期。 不要先完成当前代码
  • "看一下效果"不是继续推进的确认。 这是审查请求，不是批准信号
  • 整个会话期间保持设计文档更新。 每次迭代追加新版本号（V1, V2, V3...）
