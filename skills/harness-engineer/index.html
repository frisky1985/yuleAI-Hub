Harness Engineer

持久自治工程运行时 — 将任何仓库变成自改进的软件系统

六大核心原则

P1: 上下文工程

将上下文视为有限且宝贵的资源。积极剪裁。参见：runtime/context-engineering.md、runtime/compaction.md

P2: 工具使用

每个子 Agent 只获得它需要的工具——不多不少。参见：tools/TOOL_REGISTRY.md

P3: 验证机制

每个输出由生产者以外的人验证。参见：agents/reviewer.md

P4: 状态管理

状态存在于上下文窗口之外——在仓库中。参见：runtime/status-management.md

P5: 可观测性与反馈闭环

追踪发生了什么。将失败反馈给 harness，而不是代码。参见：runtime/observability.md

P6: 人类监督

人类审批高影响事件。Harness 显式地呈现它们。参见：runtime/autonomy-rules.md

不可妥协的规则

  1. AGENTS.md 是真相来源——每个会话先读
  2. 代码库优先于文档——冲突时信任代码
  3. 40% 上下文规则——超过 40% 前剪枝或分 Agent
  4. 无研究输出、计划、验证标准则不实现
  5. 生成和审查始终分离——不能同一 Agent
  6. 失败 = Harness 缺口——修复 harness，不只是修复症状
  7. 优化优先级：安全 → 正确 → 可靠 → 性能 → 内存 → 可维护 → 成本
  8. 最小子 Agent 范围——>5K 行按模块拆分
  9. 最大并行 5
  10. 10 分钟卡死杀死——无输出 >10 分钟则拆分重试
