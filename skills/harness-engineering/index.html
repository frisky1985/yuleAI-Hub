Harness Engineering

让现有仓库对 Agent 更可读，将关键知识转为仓库本地产物，用可执行检查替代纯文档

核心理念

把一个现有仓库逐步改造成 Harness Engineering 风格，不需要假装它在空白模板上重建。

  • 偏好仓库本地知识、可预测的结构、可执行的检查
  • 拒绝空洞的文档和隐藏的上下文

六步工作流

1. 审计当前仓库

  • 读取 AGENTS.md 和 docs/ 下的一级文档
  • 识别缺失的真相来源、死引用、占位符 CI、栈特定假设
  • 确认仓库是真实应用、治理层还是启动模板
  • 保留现有的产品和运行时结构

2. 建立最小仓库本地产物

只添加确实能改善 Agent 可读性的缺失产物：

  • docs/ARCHITECTURE.md — 架构文档
  • docs/core-beliefs.md — 核心信念
  • docs/quality-score.md — 质量评分
  • docs/observability.md — 可观测性
  • docs/product-specs/ — 产品规范
  • docs/exec-plans/ — 执行计划
  • docs/generated/README.md — 生成物索引

3. 用可执行检查替代纯策略声明

  • 优先使用 bun 编写轻量治理脚本
  • 添加可执行检查：必需文件存在、占位符过期、引用失效、契约漂移
  • 在 CI 中运行检查（每次 push 和 PR）

4. 保留通用性

  • 保留真实的项目名称、领域和运行时细节
  • 只移除偶然的一次性假设、过时的团队知识和未记录的约定
  • 文档要写得让未来的 Agent 能从仓库本身继续工作
  • 渐进式加固，而不是大范围重写

5. 让仓库对 Agent 可读

  • 每类知识放在一个明确的位置
  • 产品意图在 docs/product-specs/
  • 交付工作在 docs/exec-plans/
  • 外部库笔记在 docs/references/

6. 验证模板

  • 运行 bun run template:check 或项目等效命令
  • 修复所有硬失败
  • 警告作为待办事项
