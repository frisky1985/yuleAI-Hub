Superpowers — OpenClaw 版

Spec-first, TDD, subagent-driven 软件开发工作流。

流水线

Idea → Brainstorm → Plan → Subagent-Driven Build (TDD) → Code Review → Finish Branch

每个编码任务走这条流水线。"太简单不需要设计"永远是错的。

阶段 1：需求调研 (Brainstorming)

  1. 探索项目上下文（文件、文档、最近提交）
  2. 提问澄清问题——每次一个，优先用多项选择
  3. 提出 2–3 种方案（含权衡 + 推荐）
  4. 按部分展示设计，每部分得到批准后再继续
  5. 写设计文档 → docs/plans/YYYY-MM-DD-<topic>-design.md → 提交
⚠️ 硬门禁：设计获批前不得写任何代码

阶段 2：编写执行计划

  • 输出详细的任务级实施计划
  • 每个任务 2–5 分钟：写测试 → 看失败 → 实现 → 看通过 → 提交
  • 保存到 docs/plans/YYYY-MM-DD-<feature>.md

阶段 3：Subagent 驱动开发

每个任务的执行循环：

  1. sessions_spawn 启动实现 Agent（含完整计划上下文）
  2. 等待完成
  3. sessions_spawn 启动 spec 审查 Agent → 确认代码匹配 spec
  4. sessions_spawn 启动质量审查 Agent → 批准质量
  5. 修复问题后重新审查
  6. 所有任务完成后 → 总体代码审查 → 进入阶段 5

TDD 是强制性的：先写失败测试 → 实现 → 通过 → 提交。

阶段 4：系统化调试（bug 专线）

⚠️ 无根因调查不得修复
  1. 根因调查：读错误、重现、检查最近变更、跟踪数据流
  2. 模式分析：找工作示例、对比、识别差异
  3. 假设+验证：一次一个假设，测试证明/证伪
  4. 修复+验证：修复根因（不是症状），验证不破坏其他功能

阶段 5：完成分支

  1. 验证所有测试通过
  2. 确定基础分支
  3. 选择：本地合并 / push+PR / 保留 / 丢弃
  4. 执行选择
  5. 清理

核心原则

  • 一次一个问题 — 调研阶段每次只问一个
  • TDD 始终 — 先写失败测试，删掉测试前写的代码
  • YAGNI — 从所有设计中移除不必要的功能
  • DRY — 不重复
  • 系统化 > 即兴 — 特别是时间紧迫时
  • 证据 > 声称 — 验证后再宣布成功
  • 频繁提交 — 每次测试通过后提交
