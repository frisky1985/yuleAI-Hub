⚡ Agent Mesh Keepalive — 多层保活架构

跨 Agent 在线维持 · 4 层防护 · 自动修复 · 2026-06-13 更新

确保 Agent Mesh 中所有组件（OpenClaw、Hermes、QoderCLI、Gateway、MCP Bridge）始终在线， 会话不因进程退出或系统重启而断连。通过 4 层防护实现 99.9%+ 在线率。

四个层次从底层系统服务到上层应用逻辑，形成了完整的防御面：

层级组件作用修复方式
Layer 1 launchd / systemd OS 级守护，系统重启自动拉起 Gateway & Dashboard OS 自带重启策略
Layer 2 cron + shell 脚本 每 30 分钟健康检查，写入 JSON 状态 日志告警，不自动修复 Agent 进程
Layer 3 OpenClaw Gateway 心跳 内置心跳机制保持长连接 异常时主动通知
Layer 4 自动修复脚本 Gateway 或 Bridge 掉线时自动拉起 自动重启进程

二、Layer 1: 基础设施保活 (launchd)

2.1 Gateway 开机自启

macOS 上通过 launchd 服务实现开机自启：

# 服务文件位置
~/Library/LaunchAgents/com.openclaw.gateway.plist

# 查看服务状态
sudo launchctl list | grep openclaw
# → 33030  0  com.openclaw.gateway  (PID + 退出码 0)

launchd 的关键特性：

  • KeepAlive=true — 进程异常退出后自动重启
  • RunAtLoad=true — 用户登录时自动启动
  • WatchPaths — 配置文件变更时重载（可选）

2.2 Dashboard 自启

Gateway 内置的 Web Dashboard 随 Gateway 一起启动，地址固定：

http://127.0.0.1:18789/

2.3 MCP Bridge 文件完整性

Bridge 是 stdio 模式 — 不由 launchd 管理，而是由 OpenClaw MCP 按需自动启动。保活脚本只检查脚本文件是否存在：

ls ~/.openclaw/mcp-bridge/mcp_bridge_server.py  # 文件必须存在

三、Layer 2: 健康检查 (cron + 脚本)

3.1 cron 调度

crontab -l
# → */30 * * * * /Users/ingeek/.openclaw/mcp-bridge/agent-keepalive.sh >/dev/null 2>&1

3.2 保活脚本详解

脚本路径：~/.openclaw/mcp-bridge/agent-keepalive.sh

检测逻辑：

组件检测方式说明
OpenClawcommand -v openclaw检查 CLI 是否在 PATH 中
Hermescommand -v hermes同上
QoderCLIcommand -v qodercli同上
Gatewayopenclaw gateway status 运行时态解析输出中 "Runtime: running"
Bridge短期生成子进程请求 tools/list真正的功能测试，不是进程名匹配

完整脚本：

#!/usr/bin/env bash
set -euo pipefail

BRIDGE_DIR="${HOME}/.openclaw/mcp-bridge"
WORKSPACE="${BRIDGE_DIR}/workspace"
STATE_FILE="${WORKSPACE}/keepalive-state.json"
mkdir -p "$WORKSPACE"

now() { date -u '+%Y-%m-%dT%H:%M:%SZ'; }

check_cmd() {
  if command -v "$1" &>/dev/null; then echo "online"; else echo "offline"; fi
}

OPENCLAW=$(check_cmd openclaw)
HERMES=$(check_cmd hermes)
QODER=$(check_cmd qodercli)

GATEWAY="offline"
if openclaw gateway status 2>/dev/null | grep -q "Runtime: running"; then
  GATEWAY="online"
fi

BRIDGE="offline"
if command -v python3 &>/dev/null; then
  BRIDGE_TEST=$(python3 -c "
import subprocess, json, sys, time
p = subprocess.Popen(
    ['python3', '\$BRIDGE_DIR/mcp_bridge_server.py'],
    stdin=subprocess.PIPE, stdout=subprocess.PIPE, stderr=subprocess.PIPE,
    text=True, bufsize=0
)
time.sleep(0.2)
body = json.dumps({'jsonrpc':'2.0','id':1,'method':'tools/list','params':{}})
req = f'Content-Length: {len(body)}\r\n\r\n{body}'
try:
    out, _ = p.communicate(input=req, timeout=5)
    if 'tools' in out: print('ok')
except: pass
" 2>/dev/null)
  if [ "\$BRIDGE_TEST" = "ok" ]; then
    BRIDGE="online"
  fi
fi

cat > "\$STATE_FILE" << JSON
{
  "lastCheck": "\$(now)",
  "agents": {
    "openclaw": "\$OPENCLAW",
    "hermes": "\$HERMES",
    "qodercli": "\$QODER",
    "gateway": "\$GATEWAY",
    "bridge": "\$BRIDGE"
  }
}
JSON

echo "[\$(now)] Keepalive: O=\$OPENCLAWH=\$HERMES Q=\$QODER G=\$GATEWAY B=\$BRIDGE"

3.4 输出格式

cron 执行时将 stdout 重定向到 /dev/null，不会干扰终端。手动运行时能看到一行简洁状态：

~/.openclaw/mcp-bridge/agent-keepalive.sh
# → [2026-06-13T14:52:47Z] Keepalive: O=online H=online Q=online G=online B=online

四、Layer 3: 内置心跳 (Gateway 心跳)

4.1 OpenClaw Gateway 心跳

Gateway 配置了 30 分钟心跳间隔：

# Gateway 配置 (~/.openclaw/config.yaml)
heartbeat:
  interval: 30m         # 每 30 分钟一次
  jitter: 1m            # ±1 分钟随机

心跳触发的是 系统事件（systemEvent），不是 agentTurn：

  • 走当前主会话（main session）
  • 不会隔离上下文
  • 触发 AGENTS.md 定义的 Heartbeats 章节行为

4.2 HEARTBEAT.md 静默规则

心跳到来时按照 HEARTBEAT.md 的规则执行：

# HEARTBEAT — 周期任务（静默模式）

## 响应规则
- ✅ 首次心跳提示一次「已进入心跳静默模式」
- 🔇 之后所有心跳静默，不输出任何内容
- ⚠️ 仅在有异常时才说话（Agent 离线、Bridge 缺失、需手动干预）

这意味着除非有 Agent 掉线，否则心跳完全不可见。用户不会收到无意义的"我还活着"消息。

4.3 心跳与 cron 的配合

维度Heartbeat (Gateway)cron (agent-keepalive.sh)
触发者Gateway 内置定时器系统 crond
频率每 30 分钟（可调）每 30 分钟
作用域OpenClaw 会话保活全部 5 个组件
告警异常时说话写入 JSON + 日志
依赖Gateway 必须在运行cron 必须在运行
冗余心跳是长连接维续独立于 Gateway 运行

五、Layer 4: 自动修复

5.1 Gateway 自动修复

launchd 已在 Layer 1 提供 — Gateway 进程退出后在毫秒级自动重启。无需额外脚本。

5.2 MCP Bridge 自动修复

Bridge 是 stdio 模式，不长期驻留进程。OpenClaw MCP 按需自动拉起一个子进程，任务完成后退出。所以 Bridge 的"掉线"实际是：

  • 脚本文件缺失 → 复制或重新安装
  • Python 依赖缺失 → pip install -r requirements.txt
  • Bridge 超时 → 重试（OpenClaw MCP 自动处理）
# 手动修复 Bridge
cd ~/.openclaw/mcp-bridge
pip install -r requirements.txt
openclaw mcp list  # 确认 Bridge 正常注册

5.3 Agent CLI 掉线

如果 openclaw / hermes / qodercli 命令不可用（从 PATH 消失）：

  • 检查安装路径是否在 $PATH 中
  • 检查 Node/Python 环境是否正常
  • 重新安装：npm install -g openclaw / pip install hermes

六、手动操作指南

6.1 快速检查

# 方式一：直接运行保活脚本
~/.openclaw/mcp-bridge/agent-keepalive.sh

# 方式二：查看状态文件
cat ~/.openclaw/mcp-bridge/workspace/keepalive-state.json

6.2 在对话中检查

对 OpenClaw 说：

"检查 agent 保活状态"
"check agents status"
"看看所有 Agent 在不在线"

OpenClaw 会自动执行保活脚本并回复完整的状态。

6.3 手动触发保活检查

# 立即执行保活（不等待 cron）
cd ~/.openclaw/mcp-bridge && bash agent-keepalive.sh

# 查看最近 10 次 cron 执行
log show --predicate 'process == "cron"' --last 1h | grep keepalive

6.4 手动重启组件

# 重启 Gateway
openclaw gateway restart

# 验证 Bridge 在线
openclaw mcp list | grep bridge

# 安装/更新 Agent
npm install -g openclaw@latest
pip install --upgrade hermes
qodercli update

七、部署指南

7.1 首次安装

# 1. 确认 Agent CLI 都存在
openclaw --version && hermes --version && qodercli --version

# 2. 安装保活脚本
mkdir -p ~/.openclaw/mcp-bridge
cp agent-keepalive.sh ~/.openclaw/mcp-bridge/
chmod +x ~/.openclaw/mcp-bridge/agent-keepalive.sh

# 3. 添加 cron 任务
crontab -e
# 添加一行:
*/30 * * * * /Users/你的用户名/.openclaw/mcp-bridge/agent-keepalive.sh >/dev/null 2>&1

# 4. 验证
~/.openclaw/mcp-bridge/agent-keepalive.sh

7.2 Gateway launchd 服务安装

# 创建 launchd plist
cp ~/.openclaw/.launchd/com.openclaw.gateway.plist ~/Library/LaunchAgents/
launchctl load ~/Library/LaunchAgents/com.openclaw.gateway.plist

# 验证
sudo launchctl list | grep openclaw

7.3 故障排除

现象可能原因修复
所有 Agent offlinePATH 环境不对脚本中显式指定 export PATH=...
Gateway offlineGateway 未启动openclaw gateway start
Bridge offlinePython 环境问题pip install -r requirements.txt
状态文件不更新cron 未执行crontab -l 确认条目存在
心跳不触发Gateway 配置缺心跳检查 config.yaml 中 heartbeat 段
设计原则：保活应该是无声的。没消息就是好消息。只有在 Agent 掉线时才通知用户， 其余时间默默守护。这就是为什么心跳默认静默，cron 输出转向 /dev/null。
