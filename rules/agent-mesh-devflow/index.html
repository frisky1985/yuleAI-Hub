💡 三角色讨论法（2026-06-05 新增）

所有需求分析、优化评估、代码评审，必须先分饰三角色讨论再给方案。

角色视角关注点辩论风格
A — 系统专家系统整体功能完整性、性能边界、失效模式、时序先亮论点，防守反击
B — 架构专家软硬解耦层次边界、接口清晰、可移植性调解冲突，找折中点
C — 开发专家代码细节可读性、简洁性、异常保护、编译安全挑代码 Bug，用量化反驳

核心规则：必须相互挑战，不是轮流念稿

  • 先亮立场，再轮流转 — A 提观点 → B 或 C 立刻挑战/质疑 → A 防守或修正 → 第三方加入
  • 可以认输，不能不打 — 被驳倒后优雅地「我让步」比死扛更有用
  • Bug 用数据和场景说话 — 说「这是 Bug」时量化和重现条件
  • 逐轮收敛 — 每一轮围绕一个具体问题，结论要么共识（三票通过）、要么留待决策者拍板
  • 输出：优先级表 — 每轮结论落入 P0-P3 优先级表，P0 必须全员共识

典型讨论流程

Round 1: A提观点 → C质疑 → B调解 → 结论
Round 2: C发现Bug → A量化影响 → B分析根因 → 修复方案
Round 3: A和B辩论架构 → C提供数据 → A/B折中
Round 4: C挑战A的性能担忧 → B发现新Bug → 全员共识

真实案例：ExtFls 异步化改造讨论成果

  • P0 写校验剩余字节 Bug — 最后一步固定读 256B 越界比较（C发现，A确认影响）
  • P0 Immediate Write 与状态机冲突 — 重复发写命令（B架构分析发现）
  • P0 Cancel 不中断硬件 — SPI 仍在传输（A提出，B提出折中方案）
  • P0 WAIT_DONE 无超时 — 硬件挂起永等（A坚持，B/C最终让步）
  • P1 Erase handler break 假跳出 — for 循环内 break 只跳出内层（C指出）
  • P2 CompareHandler 未异步化（A/B建议）
  • P3 架构优化：拆枚举、下沉 Verify 到 HAL（B建议）

一、执行模式

模式命令适用场景
A. 直接 CLIhermes -z / qodercli -p日常开发、快问快答
B. 后台异步agent-delegate + task-result长任务、批量处理
C. 辅助通信context-write / context-read跨 Agent 信息同步

二、模式选择规则

if 任务预估 < 2 分钟 → 模式 A (直接 CLI)
if 任务预估 > 2 分钟 → 通知用户可选模式 B
if 需要跨 Agent 接力 → 模式 A/B + C (context 传参)

三、7 阶段流水线

需求分析(Hermes) → 排期(Hermes) → 架构(QoderCLI) → 编码(QoderCLI) → 测试用例(Hermes) → 测试执行(Hermes) → 部署(OpenClaw)

四、重试规则

每阶段最多执行 3 次。3 次失败后 OpenClaw 接管并输出异常报告。

五、总结报告

全部完成后输出：需求标题、耗时、执行模式、各阶段状态、产出文件、重试记录。
