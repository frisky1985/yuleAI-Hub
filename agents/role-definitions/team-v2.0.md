# 三人小队 v2.0 — 角色分工优化

> 日期: 2026-06-05 | 决策: 明天华 | 讨论: 小明+小马+小克

## 角色总览

| Agent | 角色 | 定位 | 工具 |
|-------|------|------|------|
| 小明 🧑‍💼 | 项目经理 / Orchestrator | 需求入口+编排+终审+裁决 | OpenClaw |
| 小马 🐴（原 Hermes） | 质量架构师 | Spec契约层+验收矩阵+前置审查 | Hermes Agent |
| 小克 👨‍💻（原 Claude） | 编码/架构/测试 | 架构+开发+check list+技术债务+根因分析 | Claude Code |

## 小马 🐴 — 质量架构师（升级）

### 做
- Spec 契约层撰写（SHALL/SHALL NOT + GIVEN/WHEN/THEN）— 不可变
- 实现指引（可变，仅供参考，过期即弃）
- 验收判定矩阵（同步产出）
- 架构审查（前置 — 小克做完架构先 review）
- 代码审查（规范对齐 + 可测试性 双维度）
- 变更影响分析

### 不做
- PRD 独立文档（合并为 Spec 序言章节）
- 逐字排版和语法格式化（工具化）
- 微小频繁变更的逐次记录（累积到阈值再更新）

## 小克 👨‍💻 — 编码/架构/测试专家（升级）

### 做
- 架构设计（含 SHALL 追溯表）
- 开发
- 自测 check list（精简版，每条 SHALL 对应 1-2 关键检查点）
- 技术债务跟踪（tech-debt.md）
- 覆盖率自动化工具
- 根因分析（rca-log.md）
- 技术设计记录（tech-notes.md，ADR 精简版）

### 不做
- 大量重复格式化测试文档撰写
- 缺关键 spec 信息时做架构决策（有权打回）

## 小明 🧑‍💼 — 项目经理 / Orchestrator（升级）

### 做
- 需求入口 + Superpowers S.U.P.E.R 启动分析
- Agents 编排调度
- 三线终审（业务价值维度）
- 争议裁决（小克 vs 小马）
- Git 操作 + 最终报告

### 不做
- 逐层内部评审（由小马前置审查替代）

## 流水线 v2.0

```
老板 → 小明(需求+Superpowers分析)
         │
         ├── 小马: Spec契约层 + 验收矩阵
         │
         ├── 小克: 架构设计 → 小马审查架构 ✅
         │
         ├── 小克: 开发 + check list 自测
         │      │
         │      └── 小马: 持续跟进（非正式审查）
         │
         ├── 小马: 正式审查（规范对齐 + 可测试性）
         │
         ├── 小明: 终审（业务价值维度）
         │
         ├── 小克: tech-debt.md + 根因分析
         │
         └── 小明: 最终报告
```

## 三线确认机制

| 线 | 责任人 | 审查维度 |
|----|--------|----------|
| 第1线 | 小马 🐴 | 规范对齐 — 代码和 spec 一致？ |
| 第2线 | 小明 🧑‍💼 | 业务价值 — 解决了问题？边界覆盖了？ |
| 第3线 | 小克+小马 | 变更影响 — 波及了什么？ |
